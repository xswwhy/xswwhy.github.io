分布式协调与同步

分布式互斥

多个应用节点同时访问临界资源,需要解决互斥问题

  • 集中式算法
  • 分布式算法
  • 令牌环算法
  • 两层结构的分布式令牌环算法

集中式算法

需要一个协调者,当有多个节点访问临界资源时,由协调者来控制多个节点的访问顺序

优点:

  1. 简单,易于实现,所有节点只与协调者通信,互相之间不通信

缺点:

  1. 协调者会成为性能瓶颈,因为所有节点都要与协调者通信
  2. 存在单点故障.协调者出现故障,会导致所有节点都无法访问临界资源

分布式算法

访问临界资源时,多个节点之间互相通信,当一个节点获取其他所有节点的同意之后,才能访问临界资源.两个节点同时想访问临界资源的话,后发起请求的会同意先发起请求的

缺点:

  1. 节点节点较多的时候,容易产生信令风暴
  2. 可用性比较低,一旦某个节点出现故障,无法发送同意,整个系统处于停滞状态,所有节点都无法访问临界资源

令牌环算法

有个令牌在所有节点之间循环传递,拿到令牌的节点才能访问临界资源.为了解决单点故障,每个节点都要记住自己在环中的位置,碰到出现故障的节点就跳过

两层结构的分布式令牌环算法

上面的三种算法都不适合规模大,节点数量多的系统,两层结构的分布式令牌环算法可以应对大规模的系统

两层结构的分布式令牌环算法与令牌环算法有点像,对所有节点进行分组,一个组组内有一个令牌环,所有组组间有一个令牌环,这就是所谓的两层结构,其实本质上还是令牌环算法的使用

分布式选举

对于一个集群来说,多个节点如何协同,如何管理呢,这个时候就需要一个主节点,那主节点是如何选出来的呢?

  • Bully 算法
  • Raft 算法
  • ZAB算法

Bully算法

每一个节点都知道其他节点的节点ID,选取节点ID最大的节点为主节点

通信过程中的消息类型

  • Election消息:发起选举
  • Alive消息:响应Election消息(不同意)
  • Victody消息:主节点向其他节点宣示主权
  1. 集群中每个节点判断自己的 ID 是否为当前活着的节点中 ID 最大的，如果是，则直接向其他节点发送 Victory 消息，宣誓自己的主权
  2. 如果自己不是当前活着的节点中 ID 最大的，则向比自己 ID 大的所有节点发送 Election 消息，并等待其他节点的回复
  3. 若在给定的时间范围内，本节点没有收到其他节点回复的 Alive 消息，则认为自己成为主节点，并向其他节点发送 Victory 消息，宣誓自己成为主节点；若接收到来自比自己 ID 大的节点的 Alive 消息，则等待其他节点发送 Victory 消息
  4. 若本节点收到比自己 ID 小的节点发送的 Election 消息，则回复一个 Alive 消息，告知其他节点，我比你大，重新选举

优点:选举速度快,简单易实现

缺点:每个节点都要有全部节点的信息,如果ID最大的节点频繁退出加入,会导致频繁切主

Raft 算法

与我们生活中的投票机制类似,一人一票,少数服从多数

节点的角色有三种:

  • Leader:当前任期获选的主节点
  • Candidate:候选者,Leader就是从候选者中选出来的
  • Follower:普通节点
  1. 初始化时,所有节点都是Follower

这里的描述较为简单,投票环节涉及任期大小与日志长度等细节没有描述,推荐一篇更详细的文章
